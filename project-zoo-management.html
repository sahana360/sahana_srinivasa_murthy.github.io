Overview

Digitising zoo operations through a full-stack web portal

Zoos have evolved from places of public entertainment to critical institutions for wildlife conservation. Managing the wealth of information about animals, staff, visitor records, and donations — historically done manually — is slow, error-prone, and vulnerable to physical data loss.

The Zoo Management System (ZMS) is a web application that automates this existing manual system using a relational database backend. It enables administrators to easily store, retrieve, update, and delete records about animals, staff assignments, visitor registrations, ticket purchases, and donations — all through a structured, access-controlled web interface.

Visitors can register, browse the animal gallery, and make donations. Admins have full dashboard control over all database tables via an authenticated portal.

Database Normalization

Normalized to Third Normal Form to eliminate redundancy

The database was systematically normalized through three normal forms to ensure minimal data redundancy, avoid update/insert/delete anomalies, and enforce clean relational structure.

1NF
First Normal Form — Atomic Values
Each table cell contains a single, atomic value and every record is unique. All fields store only one piece of information (e.g., visitor_id, username, email, password in the VISITOR table).
2NF
Second Normal Form — Full Functional Dependency
All non-prime attributes are fully functionally dependent on the primary key. Tables with composite primary keys (e.g., TICKET linking payment_id, visitor_id, animal_id) were verified to ensure no partial dependencies exist.
3NF
Third Normal Form — No Transitive Dependencies
No non-prime attribute is transitively dependent on the primary key. All non-key fields directly and exclusively describe the entity identified by their table's primary key — removing any indirect attribute dependencies.
